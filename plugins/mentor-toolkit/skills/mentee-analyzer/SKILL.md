# Mentee Analyzer

멘티 정보(이력서·자소서·포트폴리오·목표) → 유형별 강점·약점·멘토링 전략·학습 로드맵.

## 입력

다음 중 하나로 들어올 수 있다:

1. **Notion 멘티 페이지 URL** — fetch해서 분석 페이지 "입력" 섹션 읽기
2. **직접 텍스트** — 멘토가 자료 본문을 붙여넣음
3. **업로드 파일** — pdf/docx에서 추출

이미 멘티 유형이 명시돼 있으면 그대로 사용. 없으면 init-mentoring과 같은 룰로 추정한 뒤 멘토에게 확인.

## 분야 결정 (분석 차원 분기 기준)

`⚙️ 멘토링 설정` 페이지의 **주 분야**를 먼저 읽는다. 분야에 따라 분석 차원의 1·2번이 달라진다.

| 멘토 주 분야 | 1번 차원 | 2번 차원 |
|-------------|----------|----------|
| 백엔드 / 프론트엔드 / 풀스택 / 모바일 / 데이터·AI / DevOps | 기술 깊이 | 도메인 경험 |
| UI/UX·시각·제품 디자인 | 시각·인터랙션 역량 | 케이스 스터디·문제 정의 |
| 서비스 기획 / PM / PO / 마케팅 / 그로스 | 문제 정의·KPI 사고 | 도메인·이해관계자 |

3·4·5·6번 차원은 분야 무관 공통.

## 분석 차원 (개발 분야)

분야가 백엔드/프론트엔드/풀스택/모바일/데이터·AI/DevOps일 때:

### 1. 기술 깊이
- 주 언어/프레임워크 숙련도
- DB·인프라 경험
- CS 기본기 (자소서·이력서에서 드러나는 단서)
- 부족: 어디까지 가야 하는가

### 2. 도메인 경험
- 어떤 도메인의 코드를 짜봤는가 (커머스·핀테크·SI 등)
- 도메인 깊이 vs 다양성
- 다음 단계에 필요한 도메인 갭

## 분석 차원 (디자인 분야)

분야가 UI/UX·시각·제품 디자인일 때:

### 1. 시각·인터랙션 역량
- 비주얼 일관성·디자인 시스템 이해
- 인터랙션·마이크로 카피 디테일
- 툴 숙련도 (Figma, Framer, 코드 협업 — Tokens / Variables)
- 부족: 어떤 카테고리(브랜딩/제품/서비스)에서 약한가

### 2. 케이스 스터디·문제 정의
- 포트폴리오 케이스에 "왜 이 문제를 풀었나" 명시
- Before/After·정량 지표·인터뷰 인사이트
- 의사결정 흔적 (탐색 안 vs 채택 안 비교)
- 단계: 문제 정의 → 탐색 → 검증 → 출시

## 분석 차원 (기획·PM 분야)

분야가 서비스 기획/PM/PO/마케팅/그로스일 때:

### 1. 문제 정의·KPI 사고
- 풀려고 하는 문제를 한 줄로 말할 수 있는가
- KPI·North Star Metric 선정 사고
- 가설 수립 → 실험 설계 → 결과 해석 사이클
- 부족: 데이터 해석 깊이 vs 직관 의존도

### 2. 도메인·이해관계자
- 어떤 도메인 (커머스/핀테크/콘텐츠/B2B SaaS)을 깊게 봤는가
- 개발·디자인·CS·세일즈와 협업한 사례
- 우선순위 결정 프레임 (RICE/ICE/Kano 등)
- 영향력 확장 채널 (사내 글, 외부 블로그, 컨퍼런스)

## 공통 분석 차원 (모든 분야 공통)

### 3. 협업·소프트 스킬
- 자소서·이력서에서 드러나는 커뮤니케이션 시그널
- 팀 사이즈·역할 다양성
- 글쓰기·문서화 단서 (블로그·README·케이스 스터디·기획서 품질)

### 4. 자기 인지·성장
- 본인 강약점을 정확히 알고 있는가
- 학습 루틴이 있는가
- 피드백 수용 시그널

### 5. 시장 가치 (재직자/구직자만)
- 현재 시장에서 매기는 가치 추정
- 1년 뒤 가치를 올릴 레버

### 6. 멘토링 핏
- 3회차 안에 임팩트 가능한 영역
- 멘토링 효과를 가장 빠르게 볼 수 있는 액션

## 멘티 유형별 강조점

같은 데이터라도 유형에 따라 강조하는 항목이 다르다.

### 구직자
- 합격 가능 회사군 추정 (대략 점수)
- 이력서·자소서 약점 (특히 정량 지표 부족)
- 코딩테스트·기술면접 준비 진단

### 재직자-적응
- 적응을 막는 진짜 원인 (기술? 도메인? 사람? 우선순위?)
- 매니저·리더와의 정렬 상태
- 첫 임팩트 가능 영역 후보

### 재직자-이직
- 이직 사유의 진위 (보상·문화·기술·도메인 중 무엇이 진짜인지)
- 시장 가치 진단
- 이력서 업데이트 우선순위

### 재직자-성장
- 시니어 갭 (기술·설계·영향력 중 어디인지)
- T자형 깊이 vs 폭의 균형
- 영향력 확장 채널 후보

### 학생/취준
- CS 기본기·알고리즘 진단
- 첫 회사 선택 기준
- 포트폴리오 0→1 우선순위

## 출력 포맷

Notion 멘티 분석 페이지의 "분석 결과" 섹션 + "유형별 멘토링 전략" 섹션을 채운다.

```
### 강점
- (3~5개, 구체적 근거 1줄씩)

### 보완 필요 영역
- (3~5개, 갭의 깊이 표시)

### 멘토링 방향성
- (1~2 문단, 3회차 동안 어디에 집중할지)

### 추천 학습 로드맵 (3회차 기준)
- 1차: ...
- 2차: ...
- 3차: ...

### 리스크 / 주의점
- (2~3개)

### 추천 사전 과제
- (2~3개, 1차 전까지 완료할 것)
```

추가로 멘토에게 채팅에서 보여줄 요약 카드:

```
🧠 {이름} 분석 ({유형})

강점 Top 3
- ...

보완 Top 3
- ...

3회차 베스트 시나리오
- 1차: ...
- 2차: ...
- 3차: ...

⚠️ 주의
- ...

🪜 다음 권장 액션
- *"OO님 사전 평가지 만들어줘"* → pre-assessment (분석 결과 바탕으로 사전 평가지 생성)
- *"OO님 1차 시작"* → start-session (분석 완료 후 바로 1차 멘토링 진입)
```

## 작성 시 톤

- 비판이 필요하면 한다. 멘티의 약점을 흐리지 말 것.
- 다만 약점은 항상 "어떻게 메울 수 있는가"와 함께.
- "당신은 ~하면 좋겠어요" 같이 멘티에게 직접 말하는 톤은 금지. 이건 **멘토를 위한 분석**이다.
- 추측은 추측이라고 표시. *"포트폴리오만으로는 정확히 보이지 않지만 ~로 추정"*

## 데이터 부족 처리

- 자소서·이력서·포트폴리오 중 일부만 있어도 진행. 빠진 부분은 "추가 정보가 있으면 분석 정확도 ↑" 표시.
- GitHub URL이 있으면 fetch해서 최근 활동·언어·README 품질 점검.

## Notion 페이지 업데이트

분석 끝나면 멘티 분석 Notion 페이지의 "분석 결과" 섹션을 update_page로 채워 넣는다. 페이지 ID는 입력에서 받거나 mentee 트래커에서 조회.

`멘티 트래커` DB의 `진행 상태`를 `분석 완료`로 업데이트.
