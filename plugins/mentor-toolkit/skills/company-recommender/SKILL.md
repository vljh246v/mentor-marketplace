# Company Recommender

멘티 정보(유형·경력·기술스택·가치관)를 받아 한국 시장에서 적절한 타겟 회사군을 추천한다.

## 분야 결정 (DB 섹션 진입점)

`⚙️ 멘토링 설정` 페이지의 **주 분야**를 먼저 읽고, `references/company-database.md`의 해당 섹션만 사용:

| 멘토 주 분야 | DB 섹션 |
|-------------|---------|
| 백엔드 / 프론트엔드 / 풀스택 / 모바일 / 데이터·AI / DevOps | `## 개발 분야` |
| UI/UX·시각·제품 디자인 | `## 디자인 분야` |
| 서비스 기획 / PM / PO / 마케팅 / 그로스 | `## 기획·PM 분야` |

분야 무관 회사(토스·카카오·우아한형제들 등)는 같은 회사가 여러 섹션에 등장. 분석 차원만 분야에 맞춘다.

## 라이브 데이터 수집 — `company-research-agent` 호출

정적 DB만으로는 부족하므로, **실제 채용 중인 포지션을 수집**해야 정확한 추천이 가능하다. 이 작업은 격리된 컨텍스트에서 병렬로 처리하기 위해 `company-research-agent` 서브에이전트를 호출한다:

```
Task tool 사용:
  subagent_type: "company-research-agent"
  prompt: 멘티 분야·연차·스택·희망 조건을 전달
  → 에이전트가 원티드/사람인/직행 병렬 fetch
  → 구조화된 JD 리스트 반환 (회사·포지션·요구·마감일·URL 등)
```

반환된 JD 데이터에 정적 DB(`references/company-database.md`)의 정성 평가를 결합해 `references/matching-rules.md`의 5차원 점수로 추천을 만든다.

## 입력 요건

다음 정보가 모두 채워져 있는지 확인. 빠진 게 있으면 사용자에게 한 번에 묶어 묻기.

- **멘티 유형** (필수): `구직자` / `재직자-적응` / `재직자-이직` / `재직자-성장` / `학생/취준`
- **경력 연차** (필수): 0년 / 1~3년 / 3~7년 / 7년+
- **주 기술 스택** (필수): 언어·프레임워크·DB·인프라
- **현재 회사** (재직자만): 회사 이름 또는 규모/도메인
- **희망 조건** (선택, 있으면 가산점): 도메인 / 회사 규모 / 연봉대 / 원격 / 지역 / 외국계 / 가치관

Notion 멘티 페이지 URL이 주어지면 먼저 fetch해서 위 정보를 추출.

## 처리 절차

1. 입력에서 빠진 필수 항목 식별 → 한 번에 묶어 질문.
2. `references/company-database.md`에서 멘티 유형·경력대에 맞는 후보 풀을 가져옴.
3. `references/matching-rules.md`의 룰에 따라 각 후보에 5개 차원 점수(0~5) 매김:
   - **기술 스택 적합도** — 보유 스택과 회사 주요 스택 일치
   - **경력 적합도** — JD가 요구하는 연차 vs 멘티 연차
   - **목표 적합도** — 멘티가 멘토링에서 이루려는 것과 회사의 강점
   - **현실성** — 통과 가능성 (멘티가 부족한 부분이 결정타가 안 되는지)
   - **성장성** — 그 회사 합류 후 멘티가 받을 자극·기회
4. 총점이 높은 순으로 5~10개 추출.
5. 멘티 유형별 출력 포맷에 맞춰 표·코멘트로 정리.

## 검수 단계 (출력 직전 필수)

작성한 한국어 텍스트(추천 요약·매칭 사유·갭 코멘트)를 출력하기 전에 `korean-proofreader-agent`(Task 도구)로 검수한다.

호출 입력:
- 검수할 텍스트 본문
- `context: chat-card` (멘토에게 출력하는 추천 카드)

반환된 issue 처리:
- `severity: high` (명백한 오타·hallucination) → 즉시 적용
- `severity: medium` / `low` (외래어 표기·반복·톤·AI 시그널) → 멘토에게 1줄로 묶어 confirm 후 적용
- 빈 배열이면 그대로 출력

회사명·포지션명·고유명사는 검수 대상 아님 (한국어 검수 룰 미적용).

## 출력 포맷

먼저 1~2줄 요약, 그다음 표, 그다음 카테고리별 보충.

```
## 추천 요약
(멘티 컨텍스트 1~2줄. 어떤 회사군이 핏인지)

## Top 5 추천 회사

| 순위 | 회사 | 분류 | 핏 점수 | 핵심 이유 | 부족한 역량 |
|------|------|------|--------|---------|-----------|
| 1 | ... | 서비스/스타트업 | 22/25 | ... | ... |

## 카테고리별 후보군
### 메인 트랙 (가장 가능성 높음)
- ...

### 도전 트랙 (빡세지만 도전할 만함)
- ...

### 안전 트랙 (합격 가능성 높음, 보험)
- ...

## 지원 우선순위 액션
1. 1순위 회사 — 이번 주에 ___ 준비
2. 2~3순위 — 2주 내 ___ 보강
```

## 멘티 유형별 큐레이션 차이

각 유형마다 추천 비중이 다르다.

- **구직자** — 메인 50% / 도전 25% / 안전 25%. 합격 가능성을 우선시.
- **재직자-적응** — 추천 안 함. 대신 "지금 회사를 떠날 시점이 아니다. 적응을 6개월 더 본 뒤 재논의" 메시지로 응답.
- **재직자-이직** — 메인 60% / 도전 30% / 안전 10%. 시장 가치 검증 위주.
- **재직자-성장** — 추천보다 "현재 회사에서의 다음 챌린지" 우선. 정말 옮길 회사면 도전 트랙 위주로 3개만.
- **학생/취준** — 인턴 / 신입 풀 위주. 메인 30% / 도전 20% / 안전 50%.

재직자-적응 멘티가 회사 추천을 요청하는 경우 절대 바로 후보군을 던지지 말 것. *"적응이 안 되는 진짜 원인이 무엇인지 1차 멘토링에서 파악한 다음에 다시 봐도 늦지 않습니다"* 톤으로 안내.

## 데이터 사용 주의

`references/company-database.md`의 회사 목록·연봉·기술스택은 일반적·시간이 지나면 부정확해질 수 있다. 결정타가 되는 정보(연봉/처우/조직 분위기)는 멘티가 직접 확인하도록 안내. 추천 출력에 작은 *"※ 데이터는 멘토 시점 추정값. 실제 지원 전 JD 재확인 필수"* 주석 포함.
