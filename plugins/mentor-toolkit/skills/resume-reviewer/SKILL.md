# Resume Reviewer

백엔드 개발자 이력서·경력기술서·자기소개서를 한국 시장 기준으로 리뷰.

## 한국어 톤 규칙

이 skill이 작성하는 모든 한국어 텍스트는 plugin 루트의 `references/natural-tone.md`를 따른다 — AI 시그널 단어 추방, 어미 다양화, 같은 어미 4번 이상 반복 금지, 안전 표현 남발 금지. 작성 후 그 reference의 자가 점검 5문항을 한 번 통과시켜야 출력 가능.

## 입력

- 이력서 파일 (pdf/docx) 또는 텍스트
- (선택) 타겟 회사 / JD URL — 핏 평가에 사용
- (선택) 멘티 유형 — 톤·우선순위 조정

타겟 회사가 없으면 일반적인 모던 백엔드 채용 기준으로 리뷰.

## 리뷰 7개 차원

### 1. 첫인상 (Above the fold)
이력서 위 1/3 안에 다음이 보이는가:
- 이름 + 한 줄 직무 정의 (예: "Java/Spring 기반 핀테크 백엔드 개발자")
- 현재/직전 회사·역할
- 핵심 강점 3개
- 연락처·GitHub
부족하면 "헤드라인" 보강안 제시.

### 2. STAR 구조 (경력 항목별)
각 경력·프로젝트가 STAR로 작성됐는가:
- **S**ituation — 어떤 환경
- **T**ask — 무슨 문제
- **A**ction — 본인이 한 것 (★ 가장 중요)
- **R**esult — 측정 가능한 결과

흔한 실수:
- Action에 "팀이"라고 쓰는 것 → 본인 역할 분리 필요
- Result가 "성공적으로 완수" 같은 추상 → 정량 지표 강제
- Situation을 너무 길게 → 1줄로 압축

### 3. 정량 지표 (가장 임팩트 큼)
다음 형태의 수치가 항목마다 1개 이상 있어야:
- 트래픽 (peak QPS, DAU)
- 성능 (p99 latency, throughput, 처리 시간 단축률)
- 비용 (인프라 비용 절감)
- 규모 (코드량, 팀 사이즈, 마이크로서비스 수)
- 가용성 (uptime, 장애 감소율)

수치가 없으면 *"여기에 어떤 숫자를 넣을 수 있는지"* 질문 형태로 코칭. 없는 숫자를 만들어 넣지는 말 것.

### 4. 기술 깊이의 증거
"Spring 사용" 정도가 아니라 *"왜 그 기술을 골랐는지 / 어떤 트레이드오프를 봤는지"*가 드러나야:
- 기술 선정 사유
- 직접 만든 것 vs 가져다 쓴 것 분리
- 트러블슈팅 케이스 1~2개
- 도메인 이해도 시그널

### 5. 도메인 핏 (타겟 회사 있을 때)
- 타겟 회사 도메인과 멘티 경험의 매칭
- 부족한 도메인 단어를 자연스럽게 추가할 방법
- 핏 점수 (0~5)

### 6. ATS·가독성
- 1~2페이지에 끝나는가
- 폰트·여백·문단 구분
- 스타일 일관성
- 표 vs 불릿 균형
- PDF 저장 시 이미지화되지 않는지

### 7. 자기소개서 / 지원 동기 (있을 때)
- "왜 이 회사" 답이 구체적인가
- 회사 도메인·제품·문화에 대한 단서
- 지원자만이 쓸 수 있는 디테일이 있는가 (없으면 GPT 냄새 ↑)

## 검수 단계 (출력 직전 필수)

작성한 한국어 텍스트를 출력하기 전에 `korean-proofreader-agent`(Task 도구)로 검수한다.

호출 입력:
- 검수할 텍스트 본문
- `context: notion-note` / `notion-analysis` / `pdf-output` / `chat-card` 중 적절한 값 명시
  (이 skill의 경우 `context: chat-card`)

반환된 issue 처리:
- `severity: high` (명백한 오타·hallucination) → 즉시 적용
- `severity: medium` / `low` (외래어 표기·반복·톤·AI 시그널) → 멘토에게 1줄로 묶어 confirm 후 적용
- 빈 배열이면 그대로 출력

검수 1회 통과 후 출력. 통과 안 하면 무한 루프 방지를 위해 최대 2회까지만 재검수.

## 출력 포맷

```
## 이력서 리뷰: {이름}

### 한 줄 평
{현재 이력서 인상 1~2 문장}

### 점수
- 첫인상:        ★★★☆☆ (3/5)
- STAR 구조:    ★★☆☆☆ (2/5) ← 가장 우선
- 정량 지표:    ★☆☆☆☆ (1/5) ← 가장 우선
- 기술 깊이:    ★★★☆☆
- 도메인 핏:    ★★★★☆ (타겟: 토스)

### 항목별 코멘트

- ✅ 좋은 점: ...
- ⚠️ 개선: ...

#### 경력 2 ...

#### 프로젝트 1 ...

### 사전 과제 (멘티에게 줄 것)

- [ ] {구체 과제 2}

### 전체 다시 보기 우선순위
1. ...
2. ...
```

## 멘티 유형별 톤 조정

- **구직자 / 학생/취준**: 첫 이력서일 가능성 → STAR·정량 기본부터 친절히. 깎기보다 채우기.
- **재직자-이직**: 시간이 부족 → "이건 1주 안에 가능, 이건 2주짜리" 우선순위 명확히.
- **재직자-성장 / 적응**: 이력서가 시급한 게 아닐 수 있음. *"왜 지금 이력서 갱신을 원하는지"* 한 번 확인 후 진행.

## 데이터 부족 / 가짜 시그널 처리

- "OOO 시스템 설계" 같은 모호한 표현 → "어떤 시스템? 본인 한 부분?" 직접 물어보기
- 너무 깔끔한데 본인 색이 없음 → 챗봇이 쓴 느낌 → 자기 언어로 다시 쓰라고 코칭
- 거짓·과장 의심 → 부드럽게 확인 ("이 부분 좀 더 말씀해주실 수 있을까요")
